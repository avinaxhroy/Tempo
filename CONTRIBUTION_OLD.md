## Project Philosophy

Tempo is:
- **User-first**
- **Privacy-first**
- **Local-first**
- **Opinionated by design**

This means:
- Not every feature request will be accepted
- Not every pull request will be merged
- Stability and clarity matter more than feature volume

Tempo is not a generic ‚Äúcommunity playground‚Äù app. It is an evolving product with a clear identity.

## Contribution Model

### ‚úî Allowed Contribution Flow

All contributions must follow this workflow:
1. **Fork the repository**
2. **Create changes in your fork**
3. **Open a Pull Request (PR)** with:
    - Clear explanation of what you changed
    - Clear explanation of why the change is useful
    - Screenshots or examples (if UI-related)

I will review the PR and decide whether to:
- Accept it as-is
- Request changes
- Decline it

> ‚ö†Ô∏è **Opening a PR does not guarantee it will be merged.**

### ‚ùå What is NOT accepted by default

- Large architectural rewrites
- Core behavior changes without discussion
- Major UI redesigns
- Feature additions that alter Tempo‚Äôs core philosophy
- ‚ÄúExperimental‚Äù changes without a clear user benefit

**If you‚Äôre unsure ‚Äî open an issue or start a discussion first.**

## UI / UX Contributions (Important)

Tempo already has a defined visual language and interaction model.

UI/UX contributions are:
- **Optional**
- **Incremental**
- **Carefully reviewed**

### Allowed:
- Small usability improvements
- Spacing, alignment, or hierarchy refinements
- Accessibility improvements
- Minor consistency fixes

### Not allowed without prior discussion:
- Full UI redesigns
- Visual overhauls
- Changing the app‚Äôs visual identity
- Replacing existing UI patterns wholesale

> Design work done only in Figma may or may not be implemented. **Code remains the source of truth.**

## Feature Contributions

**Before building a new feature:**
1. Check existing issues‚Äîit might already be planned
2. Open an issue to discuss the idea
3. Describe the **problem** you're solving, not just the solution
4. Wait for feedback before investing significant time

Some feature requests may be deferred, scoped differently, or declined based on:
- Alignment with project philosophy
- Maintenance complexity
- Impact on app size or performance
- User experience consistency

## Decision Making

Tempo is maintained by a single owner with a clear vision for the product. This ensures:
- **Consistency**: Clear direction and cohesive experience
- **Quality**: Thoughtful review of all changes
- **Sustainability**: Manageable scope and maintenance

**This doesn't mean contributions are unwelcome**‚Äîit means they're reviewed carefully to ensure they benefit users and align with the project's goals.

If a PR is declined, it's about fit with the project direction, not a judgment of your abilities. Feedback will be provided to help you understand the decision.
